Step one: planning
and discovery

So all of our projects typically include a general "planning and discovery" phase, but a big integration project will typically require its own dedicated discovery process

It will be a lot of work and sometimes pretty painful

This slide shows some of our original Basecamps posts, Google Docs, and system documentation from InMagic

This stage took several weeks, and involved a lot of posts, and back and forth with Wilder and InMagic

  • Confirm the exact goals and requirements
  • Is there an API available? How does it look?
  • What about exposed data? (XML, CSV, JSON)
  • What documentation is available?
  • Is it better or possible to just rebuild?
  • Ask and re-ask/Read and re-read
  • Select your tools and approach (e.g. Migrate API)
  • Are the estimates still accurate?

In this stage, we learned pretty early on that the available API integration was going to be a herculean task (dedicated Windows server, which REALLY did not want to do this for our sake and for Wilder's sake

In an early meeting, we asked if there also might be some kind of export functionality that might expose the data in a machine readable format

It turns out there was, and we immediately re-focused our discovery on this process

This phase involved Wilder quite heavily in terms of effort and hours

By the end of discovery we had a pretty clear path forward, with a fairly high degree of confidence that it would work

It wasn't until completing this stage that we felt TRULY confident we could deliver what we promised, even though we had already bid on and won the project!

Step two: design and document

Once we had eliminated most of the unknowns during discovery, it was time to move into the design phase

Unlike the discovery phase, this is where we really start getting deep into the details and documentating everything down to the field level

During this stage, every detail matters

It is time to be specific, and to understand, and to write it down

  • Data analysis and documenation (InMagic)
  • Content model documentation (Drupal)
  • Experimentation and validation
  • More experimentation and validation!

This phase also involved multiple members of the Wilder team and lots of back and forth, reviewing and commenting on living documentation

This phase is really about getting all of your ducks in a row, and documenting everything from what tools you use to exactly how the external data will mapped to your website [COVER SLIDE]

We often build small proof-of-concepts during this phase also, just to make sure we are still on the right path

Step three: build

With a fully documented plan in hand, it is finally time to move into the build phase

Wilder's involvement dropped off a bit during this phase, but they were still quite heavily involved, working with InMagic to make changes to data, and working with us to verify assumptions

There were definitely changes even with careful planning and documentation during the earlier phases

  • Implement content model and views (Drupal)
  • Implement API client or automated export (InMagic)
  • Implement API consumer or automated imports (Drupal)
  • Iterate as needed
  • Testing and QA
  • Rinse and repeat until you get it right

With a good plan and good documentation, the actual build phase should be fairly straight forward [COVER SLIDE]

But even the best laid plans are no guarantee for an easy path!

We'll talk more about some of our specific struggles in a bit but for now, success!
