Step one: planning
and discovery

So all of our projects typically include a general "planning and discovery" phase, but a big integration project will typically require its own dedicated discovery process

It will be a lot of work and sometimes pretty painful but if you do it right, you win.

This slide shows some of our original Basecamps posts, Google Docs, and system documentation from InPresto

This stage took several weeks, and involved a lot of posts, and back and forth with Wilder and InPresto

Some of the things you want to uncover at this point are:

  • What are the exact goals and requirements?
  • Is there an API available? How does it look?
  • What about machine-readable export functionality? (XML, CSV, JSON)
  • What documentation is available?
  • Is it better or possible to just rebuild?
  • Ask and re-ask
  • Read and re-read
  • How much time and effort do we think this will take?

In this stage, we learned pretty early on that the available API integration was going to be a herculean task, that involved setting up and maintain a dedicated Windows server which we REALLY did not want to do for our sake and for Wilder's sake

In an early meeting, we asked if there also might be some kind of export functionality that might expose the data in a machine readable format

It turns out there was, and we immediately re-focused our discovery on this process

This phase involved Wilder quite heavily in terms of effort and hours

By the end of discovery we had a pretty clear path forward, with a fairly high degree of confidence that it would work

Step two: design and document

Once we had eliminated most of the unknowns during discovery, it was time to move into the design phase

Unlike the discovery phase, this is where we really start getting deep into the details and documentating everything down to the field level

  • Select your tools (e.g. Migrate API)
  • Data analysis and documenation (external)
  • Content model documentation (internal)
  • More Documentation!
  • Experimentation and validation
  • More experimentation and validation!

This phase also involved multiple members of the Wilder team and lots of back and forth, reviewing and commenting on living documentation

This phase is really about getting all of your ducks in a row, and documenting everything from what tools you use to exactly how the external data will mapped to your website

We often build small proof-of-concepts during this phase also, just to make sure we are still on the right path

Step three: build

With a fully documented plan in hand, it is finally time to move into the build phase

Wilder's involvement dropped off a bit during this phase, but they were still quite heavily involved, making changes to data as needed and working with us to verify assumptions

  • Implement content model and views (Drupal 8)
  • Implement API client or automated export (external/InPresto)
  • Implement API consumer or automated imports (internal/Drupal)
  • Iterate as needed
  • Testing and QA

With a good plan and good documentation, the actual build phase should be fairly straight forward

But even the best laid plans are no guarantee for an easy path!

We'll talk more about some of our specific struggles in a bit but for now, success!
